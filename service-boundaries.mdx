# Service Boundaries

## Backend API

Owns marketplace APIs, admin APIs, integrations, and orchestration.
It should not become the place where business logic leaks into clients.

## Web app

Owns the browser experience, routing, and client-side composition.
It proxies many authenticated requests through `src/web/lib/backend-proxy.ts`.

## Mobile app

Owns the mobile experience, scanner capture, and device-specific UX.

## AI Jersey Scanner

Owns jersey analysis, multi-jersey detection, title and description generation, color palette extraction, and insight extraction.
It should remain a specialized AI service, not a general marketplace API.
See [AI Jersey Scanner](/ai-jersey-scanner).

## Authentication service

Owns image-based authenticity scoring.
It stays a narrow service boundary with its own docs.
See [Authentication](/authentication).

## Supabase

Owns persistence, object storage, and database-level automation.

## Boundary checks

- If a feature needs to query shared product state, prefer the backend and database.
- If a feature needs AI analysis, route it to the scanner or auth service.
- If a feature only shapes UI, keep it in the client.
